给博客增加在线编辑能力的记录

发表于 2026-07-16
本文字数： 2k 阅读时长 ≈ 2 分钟

这次对博客做了一次小改造：我希望它不只是一个需要在本地手动改文件、手动生成、手动发布的静态页面，而是能够有一个在线编辑入口。最后采用的方案是保留 Hexo 静态博客本身，再加上 Decap CMS、GitHub OAuth 和自动部署流程。

一开始的问题

我原来的博客本质上是一个静态站点。文章写在本地的 Markdown 文件里，Hexo 负责把这些 Markdown 生成网页，然后发布到 GitHub Pages。

这种方式很稳定，也很适合个人博客，但缺点也很明显：

  1. 每次写文章都要回到本地项目里改文件。
  2. 如果换一台电脑，编辑体验就不方便。
  3. 标签、分类这些内容虽然 Hexo 支持，但也需要自己手动维护。
  4. 没有一个像后台一样的地方，可以直接新建、编辑、保存文章。

所以这次的目标不是把博客改成完整的后端系统，而是加一个轻量的在线编辑能力。

最后采用的方案

这次选择的是 Decap CMS。它不是传统意义上的数据库后台，而是一个适合静态博客的内容管理工具。

这套方案的特点

我觉得这套方案比较适合个人博客，因为它没有引入太重的后端。

它的好处是：

  1. 博客访问速度仍然是静态站的速度。
  2. 不需要自己维护数据库。
  3. 文章仍然是 Markdown 文件，长期可迁移。
  4. 可以在浏览器里编辑文章，不必每次打开本地项目。
  5. GitHub 仍然保存完整历史记录，出问题可以追溯。

它的限制也很清楚：

  1. 登录依赖 GitHub OAuth。
  2. Render 免费服务可能会休眠，第一次打开登录会慢一点。
  3. 自动发布依赖 GitHub Actions。
  4. 这不是给所有访客使用的后台，只适合自己编辑内容。

关于评论功能

这次暂时没有做评论功能。评论和在线编辑不一样，评论是给所有访客使用的交互能力，会涉及防垃圾评论、权限、存储和展示。

如果以后要做评论，我更倾向于单独处理，比如使用 Giscus 这类基于 GitHub Discussions 的方案。这样可以避免自己维护数据库，也能保留静态博客的简单性。

总结

这次改造让我对静态博客有了一个更清晰的认识：静态博客并不一定只能手动维护。它可以仍然保持静态发布，同时把编辑、授权、构建这些动作交给外部服务完成。
