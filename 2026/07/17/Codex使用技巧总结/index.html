Codex 使用技巧总结

发表于 2026-07-17
本文字数： 4.8k 阅读时长 ≈ 4 分钟

这篇文章整理一次和 Codex 使用方式有关的讨论。重点不是介绍某个单点功能，而是总结一套更高效的协作方法：什么时候直接说需求，什么时候准备 Markdown 文件，什么时候给截图、源码或本地服务，以及怎样让 Codex 在长任务里持续推进。

1. Skill 是什么，怎么调用

Codex 里有一类能力可以理解成 Skill，也就是针对某类任务预置的工作流程。例如写作前可以先构思，修 bug 时可以按调试流程推进，处理 PDF、PPT、GitHub、浏览器页面时也有对应的能力。

3. 直接写 Markdown，还是上传 .md 文件

如果内容较短，直接在聊天里写 Markdown 更方便。比如一次性问题、简单提示词、几百字的草稿、轻量需求说明，都适合直接粘贴。

如果内容较长，或者后续需要反复迭代，最好整理成 .md 文件。比如项目说明、产品需求、长期任务记录、知识库、规范文档、长提示词，都更适合放在文件里。

5. 需要登录的网站怎么处理

对于需要登录的网站，最推荐的方式是：用户自己完成登录，Codex 查看登录后的页面。

不要直接把真实密码、验证码、银行卡信息、私密 token 或管理员凭据发给 Codex。更安全的做法是：

如果是测试环境，可以提供权限最小的临时账号，用完后删除或改密。如果只是分析页面结构，也可以用截图、录屏、导出的 HTML 或复制页面内容来替代真实登录。

遇到 SSO、短信验证码、扫码登录、设备证书这类情况，通常由用户自己完成认证，Codex 再继续处理认证后的页面。

6. 内网、VPN 和 IP 限制页面

如果 Codex app 的内置浏览器运行在本机，而本机已经连接企业网络或 VPN，那么它通常会使用当前电脑的网络环境。因此，遇到 IP 白名单、内网页面或企业 VPN 限制时，内置浏览器有机会像普通浏览器一样访问。

任务类型 最好提供什么

长期项目 project-brief.md、todo.md、decisions.md
分析网页布局 截图、URL、录屏
修改网页 项目源码、HTML、CSS、JS、assets

Docker/Redis/数据库 容器配置、端口、日志、测试账号或测试密码
修 bug 现象、复现步骤、期望行为、错误日志
长任务继续推进 状态文件和当前目标

一句话总结：把 Codex 当成一个能读文件、看页面、跑命令、改代码、做验证的协作伙伴。简单任务直接说，复杂任务给文件；看不清的问题给截图，要真实修改就给源码；长任务把记忆写进 Markdown，让对话只负责推进下一步。
