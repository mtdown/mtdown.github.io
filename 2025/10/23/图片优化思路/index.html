这一个部分还是需要仔细思考，想来应该也不止是用在图片优化上，我认为在更多的地方都是可以使用这个技术的。

一、图片查询优化

首先确定什么样的数据适合缓存：

  1. 高频访问的数据：如系统首页、热门推荐内容等。
  2. 计算成本较高的数据：如复杂查询结果、大量数据的统计结果。
  3. 允许短时间延迟的数据：如不需要实时更新的排行榜、图片列表等。

Redis 分布式缓存

分布式是指将缓存数据储存在多台服务器上，以便在高并发场景下实现更高的吞吐量和容错性。也可以把同一份数据存储到多个服务器去，以实现数据的安全性.

作为分布式缓存的主流方案，REDIS具备以下几个优势：

  • 高性能：基于内存操作，访问速度极快。单节点 Redis 的读写 QPS 可达 10w 次每秒！而且Redis 的核心网络I/O和命令执行是单线程的，避免了多线程上下文切换和锁竞争所带来的巨大性能开销。——我的理解是单线程执行不是重点，这个线程运行的方式，只询问不等待才是重点。从而让Redis 具备了用一个线程同时处理成千上万个网络连接的能力。

  • 丰富的数据结构：支持字符串、列表、集合、哈希、位图等，适用于各种数据结构存储。

  • 分布式支持：高可用和高扩展可以通过 Redis Cluster 构建高可用、高性能的分布式缓存，还提供哨兵集群机制提升可用性、提供分片集群机制提高可扩展性。

    这里的可扩展性指的是系统应对“增长”的能力。可以是用户量暴增、数据量暴增、或请求量暴增。我们可以增加机器或者机器升级来解决

    高可用性指的是系统抵抗故障的能力，解决的核心手段是冗余

什么是 Redis Cluster？

Redis Cluster (集群模式) 是 Redis 官方提供的分布式解决方案。

它的核心目标是解决两个问题：

  1. **数据分片 (Sharding)**：将海量数据分散存储到多个 Redis 节点上，突破单台服务器的内存和性能瓶颈。
  2. **高可用 (High Availability)**：当集群中的某个主节点宕机时，它的从节点可以自动顶上，保证服务继续可用。

为了实现这个目标，Redis Cluster 采用了“**哈希槽 (Hash Slot)**”的设计：

  • 16384 个哈希槽：Redis Cluster 将整个数据库（逻辑上）分成了 16384 （2的14次方）个哈希槽。
  • Key 与槽的映射：当你存入一个 key 时，Redis 会用 CRC16(key) % 16384 算法计算这个 key 应该属于哪个槽。
  • 槽与节点的分配：集群启动时，这 16384 个槽会被分配给集群中所有的**主节点 (Master)**。例如，3 个主节点，可能节点 A 负责 0-5500 槽，节点 B 负责 5501-11000 槽，节点 C 负责 11001-16383 槽。
  • 高可用（主从）：每个主节点都会有至少一个从节点 (Slave) 来复制它的数据。当主节点 A 宕机，它的从节点 A1 会被集群选举为新的主节点，接管 0-5500 槽，服务不会中断。
  • 去中心化：集群中的所有节点都互相连接（通过Gossip协议），它们都知道哪个槽由哪个节点负责。当客户端连接任意一个节点并尝试操作一个不归它管的 key 时，它会告诉客户端“请重定向到 XX 节点”，整个过程对客户端是透明的。

解决缓存常见问题

使用缓存时，一般要注意下面几个问题：

  1. 缓存击穿：某些 热点数据 在缓存过期后，大量请求直接打到数据库。

​ 解决方案：设置热点数据的超长过期时间，或使用互斥锁（如 Redisson）控制缓存刷新。

  1. 缓存穿透：用户频繁请求不存在的数据，导致大量的请求直接触发数据库查询。

​ 解决方案：对无效查询结果也进行缓存（如设置空值缓存），或者使用布隆过滤器。

  1. 缓存雪崩：大量缓存同时过期，导致请求打到数据库，系统崩溃。
  1. 缓存一致性:你在“写”（增、删、改）操作时，只更新了数据库，但忘记了去处理缓存。

​ 解决方案: 先更新数据库，再删除缓存

问题 现象 解决方案
缓存穿透 查一个不存在的数据 1. 缓存空值 2. 布隆过滤器
缓存击穿 一个热点数据过期 1. 分布式锁 (Mutex) 2. 逻辑过期
缓存雪崩 一大片数据同时过期 / Redis 宕机 1. 随机过期时间 (你已在用) 2. Redis 高可用 (集群/哨兵)
